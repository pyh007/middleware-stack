# 生产事故题：促销夜搜索写入雪崩

## 场景

20:00 促销开始后，商品搜索 P99 从 180 ms 升到 8 s，写入出现 `es_rejected_execution_exception`。集群从 green 变 yellow，一台数据节点磁盘使用率 91%，老年代 GC 每分钟累计 25 秒。当天 18:30 上线了“商家自定义属性”，动态字段名直接来自商家输入；19:40 将批量写的 refresh 从 30 秒改成每条 `refresh=wait_for`。`_cat/shards` 显示一个商品主分片占全索引 63% 文档，快照最近一次成功是昨晚，但三个月未做恢复演练。

## 已知约束

- 搜索核心链路必须在 15 分钟内恢复，允许暂时隐藏新的自定义属性。
- 订单写入不依赖此集群，但商品库存同步最多允许丢 5 分钟，目标 RTO 60 分钟。
- 不能未经批准删除索引、全量重启、关闭磁盘水位或把副本永久设为 0。
- 需要保留事故现场，且所有修复必须能灰度和回滚。

## 任务

1. 写出前五分钟止损顺序，并说明每一步保护的 SLO。
2. 列出客户端、Elasticsearch、操作系统和变更层证据，区分因果与相关。
3. 判断映射爆炸、频繁 refresh、热点分片、磁盘水位、GC 和 rejected 的可能因果链。
4. 给出短期恢复、长期模型治理、容量计划、验证、灰度和回滚。
5. 设计一次满足 RPO≤5 分钟、RTO≤60 分钟的恢复演练，并定义业务校验值。
6. 按“结论→机制→证据→边界→副作用→生产验证”做五分钟面试陈述。
