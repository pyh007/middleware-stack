# 生产事故参考处置

## 结论与前五分钟

先在入口按租户和请求类型限流，暂停产生新动态字段的写入，并回滚逐条 `refresh=wait_for`；保留库存同步的受控队列，不直接丢弃。然后冻结非必要发布，记录时间线并抓取集群健康、分片、节点、线程池、任务、GC、磁盘和客户端超时。这样先降低写线程池、段生成和堆压力，同时保护库存 RPO。

## 机制与证据链

动态字段增加 cluster state 和 field metadata，超多字段提高堆与查询开销；逐条 refresh 产生大量小段，放大 merge I/O；不均匀 routing 把写入和搜索集中到一个分片；磁盘接近水位会阻止分配，副本未分配使集群 yellow；GC 和 merge 抢占资源后线程池队列填满，最终 rejected。需要用 mapping 字段数增长、segment 数/merge throttle、主分片文档与 store 偏斜、磁盘水位决策、GC 时间、queue/rejected 同一时间线证明，不能仅凭一个指标定根因。

## 恢复与边界

短期把自定义属性收敛到 `flattened` 或预定义键值结构之前先停写；恢复批量写入并使用业务可接受的 refresh interval；对异常租户限流。若磁盘余量不足，优先清理已确认可删除的过期索引或按批准扩容，不关闭水位。只在验证副本和快照后做分片迁移。force merge 不适合仍活跃的写索引。

长期使用 index template、`dynamic: strict`、字段数告警和 schema 审批；基于租户基数与流量选择 routing，监控每分片文档、store、写入和搜索偏斜；用 ILM/数据流 rollover 控制分片尺寸；容量预留故障恢复和 merge 的临时空间。变更先在回放流量验证，再按索引/租户灰度，回滚条件包括 P99、rejected、GC、分片恢复速度和数据校验失败。

## 恢复演练

在隔离集群恢复最近快照，再重放快照后 5 分钟内的可审计变更流。计时从宣布恢复到应用只读校验和小流量读写通过；校验商品总数、最大更新时间、随机商品 checksum、别名指向、关键字段 mapping、权限和库存版本。任何校验失败都不切流。若实际 RPO/RTO 超标，调整快照频率、变更日志保留、带宽和恢复并行度，并重新演练。
