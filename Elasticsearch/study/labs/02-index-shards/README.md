# 02 索引分片路由与别名

## 学习目标

- 能解释索引、主分片、副本、协调节点和 scatter/gather 的关系。
- 能证明自定义 routing 如何减少搜索分片，同时识别热点和数据局部性风险。
- 能用 write alias 和 `_aliases` 原子动作完成版本切换。
- 能从数据量、查询 fan-out、恢复速度和故障域决定分片，而不是套固定大小。

## 运行实验

```bash
make -C Elasticsearch index-shards
```

实验创建两个双主分片索引，以租户作为 routing 写入 V1；`_search_shards` 证明查询范围从两个分片缩到一个。随后把 `lab-es-orders` 写别名从 V1 原子切到 V2，并断言新文档确实进入 V2。

## 核心机制

主分片是索引期路由和扩展的基本单位，文档 ID 或自定义 routing 经哈希映射到主分片。搜索请求到协调节点后分发到相关分片，收集局部结果再 reduce。routing 降低 fan-out 的前提是读写都提供同一稳定键；低基数、超大租户或单调流量会把资源集中到少数分片。

主分片数通常不能直接在线增加。新建索引重建时，别名把物理版本与应用逻辑名解耦；一次 `_aliases` 请求可原子移除旧指向并添加新写索引。

## 观察证据

- 无 routing 的搜索包含两个 shard group，`tenant-a` routing 只包含一个。
- `_cat/shards` 展示每个主分片的文档数和节点。
- 切换后 alias 只指向 V2，通过 alias 写入的 `new-version` 能从 V2 实时 GET。

## 修改实验

把主分片改为 3，先预测两个租户落在哪些分片；再让 90% 文档使用同一 routing，比较每分片 docs。最后设计一个大租户拆桶 routing，并说明跨桶查询与迁移代价。

## 生产边界

过多小分片增加 cluster state、堆和协调开销，过大分片拖慢恢复和迁移。别名切换不自动解决增量追平、权限、模板和客户端缓存。生产变更必须校验文档数、最大更新时间、checksum、查询结果、写入水位，并保留可快速回切的旧版本。
