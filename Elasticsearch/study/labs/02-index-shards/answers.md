# 索引分片路由与别名参考要点

## 1. 写入路径

协调节点根据 routing 选择主分片，主分片校验并执行操作，再复制给 in-sync 副本，达到请求确认条件后响应。副本数与 `wait_for_active_shards`、故障和超时会改变确认边界；客户端必须处理不确定结果和幂等。

## 2. 主分片数

它决定路由空间和基础并行，过多产生元数据/堆/fan-out，过少限制水平分布且单分片过大。split 需要满足路由分片条件，shrink 要先集中分片，均有 I/O 和只读约束；通用修复仍是新建索引 reindex。用恢复速度与负载验证，不背固定 GB 数。

## 3. routing 收益

同一 routing 哈希到确定分片，查询带 routing 可只访问对应 shard group；实验从 2 组降到 1 组。未带 routing 的查询仍需 fan-out，按 routing 写入的 GET/更新/删除也必须使用同一值，否则可能找不到文档或形成不同 ID 位置。

## 4. 热点

低基数或超大租户会集中 docs、store、写入、搜索和 CPU。监控每主分片 docs/store/indexing/search/merge、节点资源和 P99；大租户可独立索引或 routing partition，但会增加跨分片查询与迁移复杂度。

## 5. 别名语义

当别名指向多个索引，`is_write_index` 明确写目标；一次原子 actions 避免先删后加的空窗或双写歧义。实验断言切换后新文档只在 V2。权限与模板按真实索引/别名组合验证。

## 6. 迁移闭环

记录起始位点，reindex 历史，双写或重放增量，暂停或做最后追平，比较 count、时间边界、checksum 与代表性查询，然后原子切换。监控新旧写入差异，保留 V1 只读和反向切换动作；过保留期再删除。

## 7. 租户设计

小租户可共享按 tenant routing，大租户独立索引/数据流或拆成多个 routing bucket，避免单分片瓶颈。结论取决于隔离 SLO、查询是否跨租户、单租户规模、成本和迁移频率。用压测与单节点故障恢复验证方案，预留大租户继续增长空间。
