# 刷新事务日志与并发控制参考要点

## 1. 四种边界

写入确认取决于主分片/副本和请求设置；实时 GET 可从引擎最新版本读取；搜索可见需要 refresh；故障恢复依赖 translog 持久化、Lucene commit、副本和故障范围。实验明确展示 GET true、count 0 到 refresh 后 count 1。

## 2. refresh/flush/merge

refresh 打开新搜索器使新 segment 可搜，flush 创建 Lucene commit 并处理 translog generation，merge 合并不可变段并回收删除。三者触发、成本和目的不同；不应以手工 flush 解决搜索不可见，也不应以 force merge 解决活跃写压力。

## 3. refresh 取舍

短 interval 降低搜索新鲜度延迟，却增加小段、文件、merge I/O 和缓存失效，可能降低写吞吐和尾延迟。批量导入可暂时延长/关闭并在批后 refresh，但要明确用户可见性和失败恢复。

## 4. 乐观并发

seq_no 区分操作先后，primary_term 防止主分片换届后的旧主/旧读写入。实验首次条件写成功，第二次复用旧值返回 409。生产记录冲突率、热点键和处理结果。

## 5. 冲突处理

计数累加或集合合并可在重新读取后按幂等规则重算；状态机跃迁、余额或人工编辑可能需要拒绝、补偿或用户合并。无脑重试会把陈旧意图覆盖到新状态，增加负载并隐藏数据争用。

## 6. 超时与幂等

超时表示结果未知，不等于失败。使用稳定文档 ID、业务版本/幂等键和读回校验，限制带退避和抖动的重试；记录请求 ID 便于服务端与客户端对账。避免自动生成新 ID 导致重复。

## 7. SLO 设计

按 500 ms 新鲜度选 refresh 并用真实流量验证 segment/merge；按 RPO 选择副本、durability、快照和可重放变更流。注入节点故障，测量确认写、搜索可见、实际丢失窗口和恢复时间；若资源或 RPO 不满足，调整拓扑而非只改一个参数。
