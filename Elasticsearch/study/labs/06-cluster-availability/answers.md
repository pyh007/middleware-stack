# 副本分配与集群可用性参考要点

## 1. 角色与状态

master-eligible 节点选主并发布 cluster state，协调节点分发/归并请求，数据节点保存分片，主分片确定写入顺序并复制给 in-sync 副本。角色可重叠，但容量与故障域要按实际职责隔离。

## 2. health

green 表示主副本全分配，yellow 表示所有主分片可用但有副本未分配，red 表示至少一个主分片缺失。yellow 索引仍可从主分片完整读写，但冗余下降，下一次故障风险更高；业务正确性仍需单独验证。

## 3. 单节点副本

同一 shard copy 不能共置，所以无第二节点时副本无法分配。改成 0 会 green，但失去节点级冗余；实验为教学恢复状态，生产应补节点/故障域或修复分配阻塞，而不是掩盖风险。

## 4. 分配决策

常见 decider 包括 same_shard、disk_threshold、filter/awareness、data tier、shards_limit 和版本兼容。按具体未分配分片调用 explain，保存 yes/no/throttle 原因；修复后观察 recovery 与资源，不能只看状态颜色。

## 5. 副本提升

master 从 in-sync copies 选择新主并发布状态；客户端可能短暂超时、重试或收到 shard unavailable。显式实验保留 es01 协调入口，停止旧主后证明另一节点成为主且文档可读。不同确认、网络分区和陈旧副本会改变 RPO。

## 6. 恢复容量

受分片字节、数量、仓库/源节点吞吐、目标磁盘 I/O、网络、并发恢复限流和 merge 影响。许多小分片增加调度与文件开销。用真实数据量演练节点丢失，记录恢复斜率、SLO 和水位，再调整分片/容量。

## 7. 可用区事故

先保护核心流量、停止无界重试和重资源任务，保存 health/shards/allocation、磁盘/恢复与故障域证据。确认剩余两区容量与 quorum，不贸然降副本或关闭水位；扩容/恢复故障区后分批恢复。验证业务读写、数据校验、恢复速率、磁盘余量和下一故障容忍，再解除限流。
