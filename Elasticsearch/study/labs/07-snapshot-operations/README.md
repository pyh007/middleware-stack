# 07 快照安全容量与生产运维

## 学习目标

- 能区分副本与快照，定义 RPO/RTO 并用恢复演练证明。
- 能创建索引级最小权限角色，验证允许与拒绝动作并完成凭据生命周期。
- 能从磁盘、heap、文档/存储量和增长率建立容量基线。
- 能设计滚动升级的前置检查、灰度指标、停止条件和回滚路径。

## 运行实验

```bash
# 安全：容量与临时只读账号
make -C Elasticsearch snapshot-operations

# 显式恢复：只删除并恢复 lab-es-07-restore
make -C Elasticsearch snapshot-restore
```

安全实验创建临时角色和用户，证明能搜索 `lab-es-07-*`、写入返回 403，并在 finally 中删除账号与角色；同时采集磁盘、heap、索引 store 和文档数。显式实验注册本地 fs 仓库，快照一个隔离索引，删除它，恢复并校验固定 checksum，最后清理快照元数据。

## 核心机制

副本同步当前状态，也会同步误删，不能提供历史恢复点。快照复用不可变段，后续快照通常只复制新增段，但删除快照需由仓库正确管理引用。快照 `SUCCESS` 不是恢复证明；版本兼容、仓库吞吐、分片数量、权限和应用校验都会影响实际 RTO。

安全由网络隔离、TLS、认证、角色最小权限、凭据轮换和审计共同构成。容量要为副本、merge、watermark、recovery 和节点故障预留，不只看当前 store。

## 观察证据

- 临时 reader 搜索命中 1，写入返回 403 security exception，实验后账号不存在。
- 容量基线包括可用磁盘、heap%、store bytes 和 docs。
- 快照状态为 SUCCESS；删除后索引 404；恢复 successful shard=1，checksum 为 `es-lab-2026`。

## 修改实验

增加第二条文档和第二个快照，比较快照内容；把角色索引模式改窄，预测读权限。写出生产恢复验证表，包括数量、最大时间、随机 checksum、mapping、别名、权限和应用冒烟。

## 生产边界

仓库凭据不可写在 compose 或源码，生产使用 keystore/secret manager 与 TLS。多个集群不能无协调地写同一仓库。升级不可假设能原地降级，回滚常需旧集群加已验证快照和流量回切。删除快照或索引必须有审批、范围、验证与恢复路径。
