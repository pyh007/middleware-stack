# 08 事故诊断与面试综合

## 学习目标

- 能把 mapping explosion、热点 routing、段/GC、线程池 rejected 与业务重试串成可证伪因果链。
- 能按业务影响、保存现场、止损、根因、修复、验证、灰度和回滚处理事故。
- 能从主分片 docs 偏斜、mapping 错误、search profile 和线程池建立证据。
- 能按“结论→机制→证据→边界→副作用→生产验证”完成面试表达。

## 运行实验

```bash
make -C Elasticsearch incident-capstone
```

实验创建三个主分片并把 75% 文档路由到同一租户，读取每主分片 docs 证明偏斜；低字段上限会拒绝新增动态字段；随后 profile 热点租户的错误查询并保存 search/write active、queue、rejected 基线。它不制造高负载，避免把压力实验混入默认安全套件。

## 核心机制

动态字段扩张 mapping/cluster state 与每段字段元数据，热点 routing 把索引和查询工作集中到一个分片；频繁 refresh/merge、堆压力和 GC 拉长服务时间后，有限线程池队列可能饱和，rejected 引发客户端重试并放大流量。这个链条只是候选假设，必须把异常/正常租户、分片和时间窗对照，证明指标变化先后。

## 观察证据

- 新增四个动态字段返回 400，外层是文档解析错误，根因是字段上限 `illegal_argument_exception`。
- 三个主分片的 docs 不均，最大值明显高于最小值。
- 热点租户的 500 查询有命中，线程池输出提供实验时的零负载基线。
- 脚本结尾要求先止住动态字段和异常流量，再决定模型或路由改造。

## 修改实验

把热点文档比例从 75% 改为 40%，预测分片 docs；再改 routing 为 `tenant-hot-0/1/2` 分桶，比较 fan-out 与均衡。不要在默认套件里加入无限并发；若做负载实验，必须另建显式命令、限制持续时间和资源并写清清理。

## 生产边界

事故中不要先全量重启、关闭水位、降副本、删索引或 force merge。先保护核心 SLO、停止重试放大并保存证据；每次只改一个主要变量。恢复后必须校验结果完整、写入新鲜度、P99、rejected、GC、磁盘和下一故障冗余。

## 完成标准

- [ ] 能从输出提出至少两个竞争假设和证伪方法。
- [ ] 能在五分钟内写出止损与证据顺序。
- [ ] 能完成 [生产事故题](../../cases/production-incident/README.md) 并对照答案复盘。
- [ ] 能把实验边界、生产副作用、灰度与回滚纳入面试答案。
