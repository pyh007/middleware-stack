# 事故诊断与面试综合参考要点

## 1. mapping explosion

结论：无界动态字段会扩大 mapping/cluster state 和字段元数据，最终影响 master、heap、写入和查询。证据要包括字段数/增长、cluster state/pending tasks、heap/GC、请求错误和发布时间。先停新增字段，再用严格模板或 flattened 重建；验证关键查询能力与 400 比例。

## 2. 热点分片

routing 低基数、超大租户、单调时间键或不均流量都会集中 workload。比较每主分片 docs/store/indexing/search、承载节点资源和同节点其他分片；若只有某分片异常是分片热点，整节点所有分片异常可能是节点资源/邻居噪声。实验给出 docs 偏斜证据。

## 3. 放大链

小段 merge 或重聚合占资源，old GC 增加服务时间，queue 填满后 rejected；无界重试提高到达率形成正反馈。用时间线、正常租户/节点对照和限流后的同步恢复证伪。扩大队列或堆可能只推迟失败。

## 4. 前五分钟

冻结发布，入口按租户/请求类型限流，停止动态字段和高代价请求，回滚已知 refresh/客户端变更，保留可重放关键写；同步保存客户端、集群、分片、节点、任务、日志和变更证据。删索引、关水位、降副本、全重启、force merge 都需评估与审批。

## 5. 根因层次

直接根因是触发 SLO 的最近机制，例如 write queue 饱和；促成因素是热点、磁盘余量不足；系统性根因是缺少 schema 门禁、容量故障模型或重试预算。通过变更/指标时间线、回滚或限流实验、对照组证明；“同时发生”不是因果。

## 6. 重建闭环

建新模板与索引，reindex 历史，双写/CDC 追平，比较 count、时间水位、checksum、mapping 和代表查询；小租户/只读灰度后原子切别名。监控错误、P99、rejected、GC、磁盘与差异，超阈值回切旧别名并停止追平写。

## 7. 六段表达

结论先限流和冻结动态字段，不用高风险动作换绿色；机制解释字段、热点、merge/GC、队列和重试链；证据列 mapping、shards、thread pool、GC/磁盘与时间线；边界指出 yellow 与数据完整性、快照恢复未知；副作用说明限流/重建影响；生产验证覆盖业务 SLO、checksum、RPO/RTO、灰度和回滚。
