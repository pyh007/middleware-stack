# 生产事故演练：支付事件积压与“Kafka 丢数”

## 场景

周五 20:00 活动开始后，`payment-events` 写入从 2k/s 升至 8k/s。20:06 告警显示某消费组总 lag 从 0 增至 120 万；消费者日志每两分钟出现一次 rebalance。20:10 业务反馈订单表只有 96 万条成功记录，而生产者报表显示发送成功 100 万条，于是有人判断“Kafka 丢了 4 万条”。

已知：Topic 有 12 个 partition、RF=3、`min.insync.replicas=2`；消费组有 24 个实例；应用自动提交开启，单条处理会同步调用超时升高的风控接口；上游重试没有稳定 `event_id`，下游唯一键冲突被统一记录为成功。最近变更是 19:55 把 `max.poll.interval.ms` 从 5 分钟降到 60 秒。

## 限制

- 不能直接 reset offset、扩分区、删除 Group 或重建 Topic。
- 订单重复扣款的风险高于短时间延迟，任何重放都必须先定义幂等键。
- 目标 RTO 为 30 分钟，事件处理 RPO 为 0；Kafka 记录至少保留 7 天。

## 任务

1. 给出前五分钟的止损动作，以及明确禁止的动作。
2. 列出生产者、Broker、Consumer Group、应用和业务库的证据链。
3. 分别验证积压、再均衡、重复、跳过处理和真实丢失五种假设。
4. 设计 30 分钟内的恢复顺序、容量保护、验证与回滚。
5. 给出长期方案：分区/实例关系、poll 与处理线程、位点策略、幂等和可观测性。
6. 用六段式回答：“Kafka 是否丢数？如何证明？”
