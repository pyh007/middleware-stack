# 参考处置：支付事件积压与“Kafka 丢数”

## 前五分钟

冻结消费者和 Topic 配置变更，记录事故时间窗、版本与实例数；保存各 partition 的 log-end、committed offset、lag、Group 状态、rebalance 原因和风控延迟。若下游继续恶化，按预案限流上游或暂停部分非关键事件，而不是删除积压。禁止在未保存位点和未验证幂等前 reset offset 或盲目重放。

## 假设验证

24 个实例竞争 12 个 partition，一半实例没有消费能力；`max.poll.interval.ms=60s` 小于最坏处理批次时间会让成员被移出组，反复再均衡进一步压低吞吐。自动提交还可能在业务完成前推进位点，形成“lag 下降但业务缺记录”的窗口。

“100 万发送成功”要核对 delivery report 的 topic/partition/offset，而“96 万订单”要区分唯一 `event_id`、上游重试、唯一键冲突和真实业务成功。逐 partition 验证目标 offset 是否存在，再用稳定 event_id 对 Kafka 记录、处理日志、DLQ 和订单表做集合对账；没有这条证据链不能宣称 Broker 丢数。

## 恢复与回滚

先恢复风控依赖或启用批准的降级；把有效消费者数降至不超过分区数，恢复经过压测的 poll 超时，并将拉取与耗时处理解耦。确认幂等键后，从事故前安全位点建立新 Group 小流量重放，比较业务结果和错误率，再逐步扩大。若重复扣款或下游饱和，立即停止新 Group，保留原 Group 位点并回滚应用版本。

长期应使用稳定 event_id、业务唯一约束和可审计的重复处理结果；采用处理成功后提交，或用 Kafka 事务原子化 Kafka 内输入/输出；监控分区级生产/消费速率、lag 增长率、处理延迟、poll 间隔、rebalance 次数、ISR 和下游错误。容量计划按最热 partition 而非总平均估算。
