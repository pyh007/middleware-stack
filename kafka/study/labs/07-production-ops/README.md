# 07 容量、可观测性、安全与恢复

## 学习目标

- 从 Broker 元数据、partition lag 和 log-dir 字节建立巡检顺序。
- 按写入率、保留、RF、故障追赶和热点估算容量。
- 理解认证、加密、ACL、配额、升级和恢复演练的生产边界。

## 运行实验

```bash
make -C kafka production-ops
```

巡检实验写入 20 条，只处理并提交 8 条，断言总 lag=12；再读取 Broker/partition 元数据和 `kafka-log-dirs` 的物理字节。恢复实验把五条业务记录导出到临时 JSONL，删除并重建隔离 Topic，重放后按业务 payload 全量校验。实验结束不会保留备份文件。

## 巡检与容量

先从业务影响和时间窗开始，再看生产/消费速率、逐 partition lag、Group 状态、ISR、请求延迟、磁盘/网络。总平均会掩盖 key 热点；容量也要按最热 Broker 和失去一台后的剩余承载能力计算。配额能隔离租户和重放流量，但限流本身会产生延迟，应有可观测告警。

## 安全与变更

本地 PLAINTEXT 只绑定 loopback。生产应使用 TLS、SASL 或平台身份、最小 ACL、短期凭据与轮换，限制管理端口并审计配置/ACL/Topic 操作。滚动升级要验证客户端协议兼容、Controller/Broker 健康、ISR 与错误率，先小范围再扩展，保留版本和配置回滚路径。

## 恢复边界

应用导出只保护 payload，不自动包含 headers、timestamp、Topic 配置、ACL、Schema、事务状态和 Consumer Group offset。生产需要跨集群复制或受支持的备份方案，并定期真实恢复。RPO/RTO 必须从业务推导，复制不能替代隔离备份，因为误删和坏数据也会被复制。

## 修改变量与完成标准

把已处理记录从 8 改成 5，先计算每分区和总 lag；恢复时故意漏掉一条，确认业务断言失败。

- [ ] 能从一次巡检输出给出诊断顺序，而不是罗列指标。
- [ ] 能计算容量并说明热点、故障、重分配和安全余量。
- [ ] 能给出认证、授权、凭据轮换、升级与审计方案。
- [ ] 能定义一个真实恢复演练的 RPO、RTO、校验和回滚。
