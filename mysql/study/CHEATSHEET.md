# MySQL 核心知识速查

这份速查表用于恢复知识地图，不代替实验。

## 表设计

- 先从访问模式、数据约束和生命周期出发，再决定表结构。
- 主键应尽量短、稳定、非空。InnoDB 聚簇索引叶子节点保存整行，二级索引叶子节点保存主键值，过长主键会放大所有二级索引。
- 金额使用 `DECIMAL`，不要使用存在二进制舍入误差的 `FLOAT`。
- 时间点优先明确时区语义。`TIMESTAMP` 会做会话时区转换且范围有限；`DATETIME` 保存字面时间。生产中常统一存 UTC，在应用层转换。
- `VARCHAR` 长度按业务上限设计；索引很长的文本前要评估存储、比较成本和前缀索引。
- `utf8mb4` 才能完整保存 Unicode。排序规则决定大小写、重音和比较行为。
- `NULL` 表示未知或不适用，不等于空字符串或零。是否允许 `NULL` 是业务约束。
- 外键提供数据库级一致性，但会增加写入检查和变更耦合；是否使用应依据一致性要求和架构约束决定。

## 索引

- InnoDB 常用索引是 B+Tree，适合等值、范围、排序和前缀匹配。
- 联合索引不是把多个单列索引简单相加。索引列顺序要由核心查询的过滤、排序和分组共同决定。
- 能使用联合索引左侧连续列的查询最容易获得稳定收益；跳过首列通常不能有效利用该索引定位。
- 二级索引找到主键后再访问聚簇索引称为回表。查询列全部可从索引获得时是覆盖索引。
- 选择性低不代表索引永远无效；是否使用取决于数据分布、返回行数、成本估算和查询目标。
- 索引会增加磁盘、缓存占用和 `INSERT/UPDATE/DELETE` 成本。上线前后都要验证读写影响。
- `LIKE 'abc%'` 通常可使用 B+Tree 前缀；`LIKE '%abc'` 通常不能用普通 B+Tree 定位。

## 执行计划

重点看：

- `type/access_type`：访问方式，不能只凭名称下结论。
- `key`：实际选择的索引。
- `rows`：优化器估算扫描行数，不是实际值。
- `filtered`：条件过滤比例估算。
- `Extra`：覆盖索引、额外过滤、临时表、排序等信息。
- `EXPLAIN ANALYZE`：真正执行 SQL，提供实际行数和耗时；不能对生产写 SQL 随意使用。

## SQL 调优顺序

1. 确认问题 SQL、频率、总耗时、业务影响和等待类型。
2. 保留 SQL、参数、表结构、索引、数据量和执行计划。
3. 先减少扫描行数和返回数据，再考虑微小语法调整。
4. 检查谓词是否可索引、联合索引顺序、回表、排序、临时表和分页方式。
5. 用相同数据和参数验证优化前后结果一致、执行计划稳定。
6. 评估写入放大、锁、磁盘和缓存影响，准备回滚方案。

常见问题：

- 对索引列使用函数或隐式类型转换，可能失去高效范围定位。
- 深分页 `LIMIT offset, size` 仍要扫描或跳过前面的行；连续翻页优先考虑基于稳定排序键的游标分页。
- `SELECT *` 增加网络、内存和回表成本，也使接口更容易被表结构变化影响。
- 大事务会延长锁持有时间、增加 undo 和复制压力。

## 事务与 MVCC

- `READ COMMITTED` 通常每条一致性读创建新 Read View；`REPEATABLE READ` 通常复用第一次一致性读建立的视图。
- 普通 `SELECT` 通常是快照读；`UPDATE/DELETE/SELECT ... FOR UPDATE` 是当前读，需要读取最新满足条件版本并加锁。
- MVCC 依赖 undo 版本链和可见性判断，不等于所有读都不加锁。
- 事务中避免远程调用、用户等待和大批量无关计算，尽量缩短连接、锁和历史版本占用。
- 并发扣减优先考虑带业务条件的原子更新；悲观锁和乐观版本号都要定义失败与重试语义。

## 锁与死锁

- InnoDB 锁基于索引记录和范围；缺少合适索引可能增加扫描、锁冲突和持有时间。
- 记录锁保护已有索引记录，间隙锁保护记录间范围，临键锁通常是记录与前间隙的组合。
- 从 `performance_schema.data_lock_waits` 沿阻塞方向找到根阻塞事务，根阻塞连接可能正在 `Sleep`。
- 死锁存在等待环，InnoDB 会回滚一个完整事务；应用应从事务边界有限重试并保证幂等。
- 统一资源访问顺序、缩短事务、补充索引和减小批次可以降低死锁概率。

## InnoDB 与日志

- buffer pool 缓存数据页；修改先产生脏页，后台再刷新数据文件。
- redo 使用 WAL 支撑已提交修改的崩溃恢复；undo 支撑回滚和 MVCC；binlog 服务复制和时间点恢复。
- redo 与 binlog 分属不同层，两阶段提交用于协调两者的事务状态。
- `innodb_flush_log_at_trx_commit=1` 与 `sync_binlog=1` 是常见强持久性组合，但完整保证还依赖操作系统与存储。
- 状态变量应比较时间窗口增量；高 buffer pool 命中率也不能单独排除 IO 问题。

## 复制与恢复

- RPO 定义可接受的数据丢失窗口，RTO 定义可接受的恢复时间。
- 复制会传播误删，不能替代独立、异地、受权限保护且经过恢复验证的备份。
- GTID 简化副本定位缺失事务和拓扑切换，但不会自动保证无损故障转移。
- 副本延迟时会出现旧读，写后读需要明确路由或一致性等待策略。
- PITR 通常需要可恢复基线加连续完整的 binlog；恢复到新实例后先校验再处理生产数据。

## 生产运维

- 同时观察业务、连接、查询、锁、存储、复制恢复和安全，不能只看 CPU。
- SQL 优先级应结合总耗时、频率、业务价值和尖峰风险，而不是只看单次最慢。
- 在线 DDL 仍可能产生元数据锁、IO、临时空间和复制延迟。
- 连接池大小是全局预算：应用副本数乘以单池上限不能超过实例安全并发。
- 所有变更都要有基线、容量、灰度指标、终止阈值和可执行回滚。

## 面试回答结构

```text
结论 → 底层机制 → 场景或实验依据 → 适用边界 → 副作用 → 生产验证
```

不要背“口诀式绝对结论”。例如“范围条件之后的索引列一定失效”“区分度高的列永远放最前”都忽略了具体查询、MySQL 版本和优化器行为，应以执行计划和数据分布验证。
