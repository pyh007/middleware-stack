# 综合案例：大促期间订单数据库故障

## 背景

20:00 活动开始后，订单接口 P99 从 80ms 上升到 8s，部分请求超时。监控同时显示：

- 应用连接池等待增加，MySQL `Threads_connected` 接近上限。
- `Threads_running`、CPU 和磁盘读取上升。
- 大量订单查询使用 `DATE(created_at)` 并深分页。
- 一个后台批处理开启事务后调用外部接口，持续数分钟。
- 副本延迟达到 15 分钟，但部分写后读请求仍被路由到副本。
- 最近一次恢复演练是半年前，当前备份任务显示“成功”。

## 任务

按时间顺序写出你的处理方案：

1. 前 5 分钟如何止损，哪些动作不能贸然执行？
2. 需要保存哪些数据库与应用证据？
3. 如何识别总成本最高的 SQL、根阻塞事务和复制延迟原因？
4. 短期如何处理深分页、长事务和写后读？
5. 长期索引或 SQL 方案是什么，如何安全上线？
6. 如何验证数据一致性，以及备份是否真的可以恢复？
7. 给出监控、容量、权限和演练方面的预防措施。
