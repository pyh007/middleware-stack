# 01 表设计与数据类型

## 学习目标

完成本模块后，你应该能：

- 从业务约束和访问模式出发表达表设计，而不是只背三范式。
- 解释主键为什么会影响所有 InnoDB 二级索引。
- 正确选择金额、时间、状态、文本和业务编号的数据类型。
- 说明唯一约束、检查约束、外键的价值和生产取舍。
- 识别“所有字段都是字符串”“金额使用浮点数”等常见问题。

## 开始前先回答

不要先看答案，打开 [exercises.md](exercises.md) 口述第 1～3 题。不能讲清楚很正常，它们就是本次实验的观察目标。

## 运行实验

在仓库根目录执行：

```bash
make -C mysql table-design
```

实验会重建三张 `lab_` 表，不影响现有快速示例：

- `lab_bad_orders`：故意使用浮点金额、字符串日期和可变文本状态。
- `lab_design_users`：展示主键、唯一约束、字符集和时间字段。
- `lab_design_orders`：展示业务编号、精确金额、状态检查和外键。

## 观察重点

### 1. 类型表达业务约束

错误表允许写入 `2025-99-99` 这样的日期，因为数据库只知道它是字符串。正确类型不仅节省空间，还让比较、排序、函数和约束具有明确语义。

### 2. 技术主键与业务编号分离

`id` 用作稳定、较短的聚簇索引键；`order_no` 使用唯一约束表达业务唯一性。业务编号规则以后变化时，不需要级联修改所有引用关系。

这不是“永远使用自增主键”的绝对规则。分库全局 ID、写入热点、信息暴露等因素都可能改变选择，但必须说明取舍。

### 3. 约束是最后一道防线

实验会尝试写入非法订单状态，并由存储过程捕获错误。应用层校验改善体验，数据库约束保护所有写入入口，两者不是互斥关系。

### 4. 表设计必须结合查询

范式解决依赖和一致性问题，却不能独立决定最终模型。还要考虑核心查询、更新频率、数据生命周期、冷热分层和容量。冗余字段可以是有意识的性能设计，但必须定义同步方式。

## 修改实验

至少完成一项：

1. 把 `order_no` 攓成主键，比较 `information_schema.statistics` 中二级索引的设计影响。
2. 给用户增加软删除字段，思考“删除后能否重新注册同一邮箱”。
3. 增加订单明细表，决定商品名称和成交价是否需要冗余。

## 完成标准

- 能在三分钟内评审一张订单表的主键、金额、时间、状态和约束。
- 能解释 `FLOAT` 与 `DECIMAL`、`DATETIME` 与 `TIMESTAMP` 的差别和边界。
- 能说明外键在一致性、写入性能、变更和分库场景中的取舍。
- 不看答案完成 [exercises.md](exercises.md) 至少 4 题。
