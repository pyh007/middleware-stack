# 02 索引原理与设计

## 学习目标

- 解释 InnoDB 聚簇索引、二级索引、回表和覆盖索引。
- 根据过滤、排序和返回列设计联合索引，而不是机械套用口诀。
- 用 `EXPLAIN` 与 `EXPLAIN ANALYZE` 验证扫描行数和实际耗时。
- 识别无效、重复和收益不足的索引，并说明索引的写入成本。

## 运行实验

在仓库根目录执行：

```bash
make -C mysql index
```

命令会先重建 10 万行 `lab_orders`，再依次观察：

1. 没有合适索引时的全表扫描。
2. `(customer_id, status, created_at)` 联合索引的过滤和排序效果。
3. 跳过联合索引首列时，优化器如何选择访问方式。
4. 等值条件与范围条件对后续排序列的影响。
5. 覆盖索引与读取非索引列时的差异。

## 不要把口诀当结论

### 最左匹配

联合索引按完整键值有序。查询能确定左侧连续前缀时，MySQL 容易缩小扫描范围；跳过首列，后续列通常不能用于直接定位。但 MySQL 可能选择全索引扫描、跳跃扫描或其他访问方式，所以最终以具体版本、统计信息和执行计划为准。

### 范围条件

某列进入范围扫描后，后续列可能仍参与索引条件下推，但通常不能继续缩小同一连续索引范围；能否同时避免排序也要结合等值前缀和排序方向判断。不要简单回答“范围之后全部失效”。

### 覆盖与回表

InnoDB 二级索引叶子节点包含主键。查询所需列全部在二级索引中时，可以直接返回结果；如果还读取 `note` 等非索引列，就需要按主键访问聚簇索引。覆盖索引提高读性能，但会放大索引和写入成本。

## 修改实验

至少完成两项：

1. 把联合索引改成 `(status, customer_id, created_at)`，比较三类查询。
2. 把目标客户改成返回几千行的条件，观察回表成本变化。
3. 删除普通联合索引，只保留覆盖索引，判断它是否构成冗余。
4. 增加 `ORDER BY amount`，预测并验证是否出现额外排序。

## 完成标准

- 能画出聚簇索引和二级索引叶子节点保存的内容。
- 能根据三条核心 SQL 设计索引顺序，并说明没有服务哪些查询。
- 能从 `key`、`rows`、`actual rows`、`Extra` 判断索引收益。
- 能说出新增索引对写入、缓存、磁盘和 DDL 的副作用。
