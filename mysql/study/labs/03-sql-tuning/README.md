# 03 SQL 调优

## 学习目标

- 建立“先确认影响和证据，再改 SQL 或索引”的排查顺序。
- 用 `EXPLAIN ANALYZE` 区分估算行数、实际行数和执行耗时。
- 识别索引列函数、深分页和过度取列等典型问题。
- 验证优化前后结果一致，并评估写入和上线风险。

## 运行实验

```bash
make -C mysql tuning
```

实验会重建 10 万行订单，然后展示：

1. `DATE(created_at)` 与半开时间范围的扫描差异。
2. `LIMIT 90000, 20` 与基于稳定游标的分页差异。
3. 选择必要列与覆盖索引的关系。
4. 本地慢查询日志如何提供频率、耗时和扫描行数证据。

`EXPLAIN ANALYZE` 会真实执行查询。学习环境可以直接使用；生产环境中，未知 SQL 尤其是写 SQL，必须先评估风险或在副本、影子数据上分析。

## 调优闭环

```text
确认业务影响
→ 捕获 SQL、参数、频率和等待
→ 保存表结构、索引、数据分布和执行计划
→ 提出最小改动
→ 验证结果一致与性能变化
→ 评估写入、锁、缓存和复制影响
→ 灰度、监控、回滚
```

单次执行从 100ms 降到 10ms 很好，但一条每秒执行十万次的 2ms SQL 可能更值得处理。调优要看总资源消耗和业务影响，而不是只盯最慢的一次。

## 修改实验

1. 把时间查询改成 `YEAR(created_at)=2025`，预测扫描范围。
2. 给 `created_at` 建函数索引或生成列，比较它与改写范围条件的维护成本。
3. 把游标分页的排序改为不唯一列，制造重复或遗漏，再加入 `id` 作为稳定次序。
4. 将数据量缩小到 100 行，观察优化器选择是否改变。

## 完成标准

- 能独立使用慢日志和执行计划定位一个慢 SQL。
- 能解释为什么半开区间通常适合按天查询。
- 能说明深分页为什么慢、游标分页有什么产品限制。
- 优化报告包含结果一致性、性能证据、副作用、上线与回滚方案。
