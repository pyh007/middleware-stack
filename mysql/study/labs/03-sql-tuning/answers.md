# SQL 调优练习参考要点

## 1. 先收集证据

确认发生时间、持续时间、受影响接口、吞吐和延迟变化，再看数据库 CPU、IO、连接、锁等待和复制状态。对目标 SQL 保存完整语句、真实参数、调用频率、总耗时、返回行数、表结构、索引、数据量和执行计划。

## 2. EXPLAIN 与 EXPLAIN ANALYZE

`EXPLAIN` 主要展示优化器估算的计划，不执行普通查询；`EXPLAIN ANALYZE` 会真实执行并给出实际行数、循环和时间。后者对重查询会真实消耗资源，对写语句还可能改变数据，不能在生产未知风险下随意运行。

## 3. 索引列函数

普通 `created_at` B+Tree 按原值有序。对每行计算 `DATE(created_at)` 后再比较，通常无法直接定位原值范围。改成原列上的半开范围可做范围扫描。函数索引或生成列也是候选，但增加存储和写入维护成本。

## 4. 半开时间区间

半开区间不依赖字段精度，能自然覆盖毫秒、微秒，并且相邻日期没有重叠或缝隙。`23:59:59` 在高精度字段上会漏掉当天最后一秒中的记录。

## 5. 深分页

数据库仍需按顺序找到并丢弃 offset 之前的记录，offset 越大扫描越多。游标分页从上一页稳定排序键继续，复杂度更稳定，但不方便随机跳页；数据并发变化时还要定义一致性，排序键必须稳定且最好唯一。

## 6. SELECT 星号

它增加磁盘读取、回表、网络传输、序列化和内存成本，阻碍覆盖索引，还让调用方隐式依赖表结构和列顺序。确实需要整行且行很窄时未必是主要问题，仍需数据验证。

## 7. 索引后仍慢

检查返回量、回表、排序和临时表、连接和锁等待、统计信息与数据倾斜、缓存冷热、磁盘、并发量、网络、查询次数以及 SQL 是否选中目标索引。慢可能不在扫描路径，也可能是总调用量造成。

## 8. 安全上线

在代表性数据和参数上验证结果集一致、执行计划和延迟改善，同时压测写入、磁盘、缓存、锁和复制延迟。明确 DDL 算法、发布窗口、灰度指标和回滚步骤，上线后观察一个完整业务周期。

## 9. 优化优先级

不一定。通常结合总耗时、调用次数、扫描行数、资源占用、业务重要性和优化成本排序。偶发离线查询可能比每秒执行数万次的小查询优先级低，但也要考虑它是否会造成资源尖峰或阻塞。
