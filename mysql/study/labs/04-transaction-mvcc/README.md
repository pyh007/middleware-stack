# 04 事务、隔离级别与 MVCC

## 学习目标

- 解释事务的原子性、一致性、隔离性和持久性分别由什么机制支撑。
- 区分脏读、不可重复读、幻读和丢失更新。
- 理解 Read View、undo 版本链、快照读与当前读。
- 结合业务一致性和并发成本选择隔离级别与事务边界。

## 运行实验

```bash
make -C mysql mvcc
```

脚本会创建两个真实数据库连接：事务 A 保持打开，事务 B 修改并提交，再由事务 A 重新读取。你会观察到：

- `READ COMMITTED` 下两次快照读可以看到不同的已提交值。
- `REPEATABLE READ` 下普通查询继续看到原快照。
- 同一个 `REPEATABLE READ` 事务使用 `SELECT ... FOR UPDATE` 时执行当前读，看到最新已提交版本。

## 核心机制

MVCC 主要通过隐藏事务信息、undo 版本链和 Read View 判断某个版本对当前事务是否可见。它让普通一致性读避免给记录加共享锁，但不代表事务中所有语句都不加锁。

`UPDATE`、`DELETE`、`SELECT ... FOR UPDATE`、`SELECT ... FOR SHARE` 属于当前读，需要读取满足条件的最新版本并进行锁定。快照读与当前读混用时，必须理解它们可能看到不同版本。

## 生产边界

- 事务越长，锁、undo、历史版本和连接占用时间通常越长。
- 不要在事务中等待用户输入、调用慢远程接口或做无关计算。
- 隔离级别不能自动解决所有并发业务问题；库存扣减等场景还要使用条件更新、锁或乐观版本号。
- 重试死锁或序列化失败时，必须保证整个事务可安全重试，而不是只重试最后一条 SQL。

## 完成标准

- 能画出两个事务、提交点、Read View 和可见版本的时间线。
- 能解释为什么同一事务中的普通查询和 `FOR UPDATE` 可能读到不同值。
- 能根据账户余额、库存和报表场景说明隔离级别取舍。
- 不看答案完成至少 5 道练习题。
