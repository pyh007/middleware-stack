# 事务与 MVCC 参考要点

1. 原子性主要依赖 undo 和事务管理；隔离性依赖锁与 MVCC；持久性主要依赖 redo、刷盘策略和崩溃恢复；一致性是约束、应用规则以及前三者共同作用的结果，不能简单归到某一个日志。
2. `READ COMMITTED` 通常每条一致性读语句创建新的 Read View；`REPEATABLE READ` 通常在事务第一次一致性读时建立并复用，因此对后来提交的版本可见性不同。
3. 普通查询通常是快照读，按 Read View 选择历史可见版本；`FOR UPDATE` 是当前读，要读取并锁定最新满足条件的版本。
4. 不是。锁定读、写操作、DDL 和部分约束检查仍会加锁；快照读也可能等待元数据锁等其他资源。
5. InnoDB 的当前读配合 next-key lock 能防止许多范围幻读，但快照读与当前读混用、业务先判断后写等情况仍需具体分析，不能用隔离级别名称代替业务正确性证明。
6. 远程调用会延长事务和锁持有时间，增加超时、死锁、连接占用和不确定性。通常先缩短数据库事务，再通过幂等、消息或补偿协调外部系统。
7. 这是典型读改写竞争，可能丢失更新或超卖。可使用 `UPDATE ... SET stock=stock-1 WHERE stock>0` 的条件原子更新、`FOR UPDATE` 悲观锁或带版本号的乐观锁。
8. 不正确。死锁牺牲者的整个事务已回滚，应从事务边界重新执行，并保证操作幂等、限制重试次数和加入退避。
