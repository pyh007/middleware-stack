# 05 锁、锁等待与死锁

## 学习目标

- 区分记录锁、间隙锁、临键锁、意向锁和元数据锁。
- 使用 `performance_schema.data_locks`、`data_lock_waits` 和 InnoDB 状态定位阻塞链。
- 稳定复现死锁，理解为什么 InnoDB 必须回滚其中一个完整事务。
- 从访问顺序、索引、事务范围和重试机制降低锁冲突。

## 运行实验

```bash
make -C mysql locks
```

实验分两部分：

1. 事务 A 更新一行不提交，事务 B 更新同一行进入等待；脚本从 `performance_schema.data_lock_waits` 读取请求者和阻塞者。
2. 两个事务以相反顺序更新 `id=1` 和 `id=2`，形成循环等待；InnoDB 检测后回滚其中一个事务，并留下最近死锁报告。

## 关键判断

锁问题排查不是看到“慢”就扩大超时。先回答：

- 谁在等待谁？阻塞者当前是在执行还是空闲但事务未提交？
- 锁住的是记录、范围还是表结构？
- SQL 是否使用了合适索引，实际扫描和锁定了多少记录？
- 两个事务能否统一访问顺序？事务中是否包含远程调用或批量处理？

死锁是并发控制的正常可能结果，不等同于数据库损坏。正确做法是缩小发生概率，并让事务可以有限、带退避地整体重试。

## 索引与锁范围

InnoDB 锁的是索引记录和索引范围。缺少合适索引时，SQL 可能扫描并检查大量记录，锁影响范围和持有时间都可能扩大。`REPEATABLE READ` 下范围当前读还可能使用 next-key lock 防止范围内插入。

## 完成标准

- 能从阻塞链找到根阻塞事务，而不是只处理最末端等待者。
- 能解释死锁检测、锁等待超时和业务重试的差别。
- 能给出统一加锁顺序、缩短事务、补充索引三类修复方案及副作用。
- 不看答案完成至少 6 道练习题。
