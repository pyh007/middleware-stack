# 锁与死锁参考要点

1. 记录锁保护索引记录；间隙锁保护索引记录之间的范围而不是已有行；临键锁通常是记录锁与其前间隙的组合。实际范围受隔离级别、访问方式、唯一索引和谓词影响。
2. 不一定。它们可能更新相同的二级索引记录、被外键或唯一约束检查关联，也可能因之前语句、范围锁或元数据锁而阻塞。
3. 从等待边连接请求事务和阻塞事务，沿 blocking 方向继续查找，直到找到没有继续等待的事务。根阻塞者可能当前没有执行 SQL，只是连接处于 `Sleep` 且事务未提交。
4. InnoDB 基于索引访问和加锁。扫描更多索引记录会增加检查、锁定或冲突机会；即使最终只修改少量行，也可能持锁更久。
5. 死锁存在循环依赖，InnoDB 可以立即检测并回滚牺牲者；普通锁等待没有环，只能等待持有者释放或达到 `innodb_lock_wait_timeout`。
6. 从完整事务边界重试。牺牲者事务已整体回滚，只补执行最后一条 SQL 可能破坏业务原子性。
7. 所有事务按相同顺序获取资源时，不容易形成 A 等 B、B 又等 A 的环，但仍需控制事务大小和索引范围。
8. 可能。`REPEATABLE READ` 的范围当前读可能对不存在值所在间隙加锁，从而阻止其他事务在该范围插入；唯一索引等值查找等情况行为可能不同。
9. 查看 `performance_schema.metadata_locks`、当前语句和事务，寻找长期未提交事务或正在使用对象的会话。不要只反复重试 DDL，应先处理根阻塞并评估终止事务的业务风险。
