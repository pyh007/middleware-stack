# InnoDB 与日志参考要点

1. 数据页通常比一次修改大且写入位置分散。WAL 先顺序写较小的 redo，再异步刷新脏页，能降低提交延迟；崩溃后通过 redo 恢复已提交修改。
2. redo 是 InnoDB 层用于崩溃恢复的物理变化日志，空间循环使用；binlog 是 Server 层的逻辑变更日志，按文件追加，主要用于复制和恢复。两者生命周期与格式不同。
3. undo 还用于构造 MVCC 历史版本，让其他事务按 Read View 读取旧值。
4. 内存中已修改但尚未刷新到数据文件的页是脏页。存在脏页是正常机制；要结合增长速度、checkpoint、刷页能力、延迟和可用干净页判断压力。
5. 常见配置下，事务提交的 redo 和 binlog 都要求同步到持久化存储后再返回，尽量保证操作系统或 mysqld 崩溃时不丢已确认事务；硬件、文件系统和存储缓存仍是保证链的一部分。
6. 避免 redo 已提交而 binlog 缺失，或 binlog 已记录而 InnoDB 最终回滚，从而造成主库状态与复制/恢复链路不一致。
7. 不一定。总请求量巨大时很小的未命中比例也可能带来大量 IO；写入刷脏、binlog、临时文件和其他进程也可能是瓶颈。
8. 活跃 Read View 仍可能需要旧版本，purge 不能删除它们，导致历史链、undo 空间和查询成本增长。
9. 先根据 redo 恢复页面到一致状态，再识别事务状态：已提交事务保留，未提交事务利用 undo 回滚。具体过程还包括检查点和日志扫描。
