# 07 备份恢复与主从复制

## 学习目标

- 用 RPO、RTO 定义恢复目标，而不是只说“每天备份”。
- 区分逻辑备份、物理备份、全量、增量和 binlog 时间点恢复。
- 实际完成备份、误删、恢复和业务校验。
- 理解异步复制、延迟、GTID、只读副本以及复制不等于备份。

## 备份恢复实验

```bash
make -C mysql backup-restore
```

脚本会：创建三条确定数据、使用 `mysqldump --single-transaction` 备份、删除全部数据、恢复备份，并验证记录数和业务内容。备份文件保存在临时目录，实验结束自动清理。

一次恢复演练必须验证业务数据，而不只是看到恢复命令返回 0。真实系统还要验证约束、账号权限、存储过程、事件、字符集以及应用能否正常启动。

## 独立 GTID 主从实验

```bash
make -C mysql replication
```

该命令会启动两个额外的 MySQL 8.4 容器：

- 主库：`127.0.0.1:3307`
- 副本：`127.0.0.1:3308`

脚本创建复制账号、配置 GTID 自动定位、在主库写入两条事件，并从副本读取验证。学习完成后清理：

```bash
make -C mysql replication-down
```

`replication-down` 会同时删除这套独立实验的命名卷，便于下次从空环境重建；不会影响端口 3306 的主学习数据库。

## 生产边界

- 复制会忠实传播误删、错误更新和部分逻辑破坏，因此不能替代独立备份。
- 副本延迟时，读写分离可能读到旧数据；写后读一致性需要路由、等待位点或业务降级策略。
- `mysqldump --single-transaction` 适合 InnoDB 一致性快照，但长时间导出会维持 Read View，也要关注 DDL、undo 和存储压力。
- PITR 通常需要一个可恢复全量基线，加上从基线位置开始连续、完整且经过校验的 binlog。
- 备份必须异地、加密、限制权限，并定期真实恢复；“任务成功”不代表“数据可恢复”。

## 完成标准

- 能从业务给出合理 RPO/RTO，并反推备份频率和恢复架构。
- 能独立恢复误删表并验证业务内容。
- 能解释 GTID、IO 接收线程、SQL 应用线程和复制延迟。
- 能说明主从切换前必须确认的数据一致性、写入口和回切问题。
