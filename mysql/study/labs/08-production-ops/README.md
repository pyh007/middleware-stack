# 08 生产巡检、变更与综合验收

## 学习目标

- 建立配置、连接、SQL、锁、存储、复制、备份和安全的监控地图。
- 使用时间窗口和业务基线解释指标，而不是看到单个数值就报警。
- 设计数据库变更的容量评估、灰度、监控和回滚。
- 面对综合事故时先控制影响、保存证据、定位根因，再长期修复。

## 运行巡检实验

```bash
make -C mysql production
```

命令会准备 10 万行订单数据，然后采集：

- 连接、超时、SQL 模式和慢日志配置。
- 当前连接分布与历史连接失败累计值。
- 按总耗时排序的 SQL 摘要。
- 表与索引容量估算。
- 尚未被统计使用的索引候选。
- 一个临时只读账号的最小权限授权。

`schema_unused_indexes` 只反映统计开启后的观测，可能没有覆盖月末、季度或容灾任务。它只能生成候选清单，不能直接驱动删除。

## 生产观测地图

- 业务：吞吐、成功率、平均与尾延迟、核心功能可用性。
- 连接：连接数、运行线程、连接失败、连接池等待和泄漏。
- 查询：总耗时、频率、扫描/返回比、临时表、排序和慢日志。
- 并发：锁等待、死锁、长事务、undo 历史、元数据锁。
- 存储：IO 延迟、磁盘利用率、buffer pool、redo、刷脏和空间增长。
- 复制恢复：复制线程、延迟、错误、binlog 保留、备份和恢复演练。
- 安全：账号生命周期、最小权限、TLS、审计、敏感数据和密钥轮换。

## 变更检查

任何索引、DDL、参数或版本变更都应回答：

1. 解决什么可量化问题？基线证据是什么？
2. 数据量、增长率、临时空间和执行时间是多少？
3. 是否产生元数据锁、IO 峰值、复制延迟或执行计划变化？
4. 如何灰度，观察哪些成功与终止指标？
5. 回滚是否真的可执行，预计需要多久？

## 综合验收

完成 [生产事故综合案例](../../cases/production-incident/README.md)。报告必须串联慢 SQL、连接堆积、锁等待、复制延迟和恢复风险，不能只给“加索引”一个答案。

## 完成标准

- 能在 15 分钟内列出数据库故障的现场采集清单。
- 能根据 SQL 总成本和业务价值排序优化优先级。
- 能写出一次安全 DDL 的容量、锁、复制、监控和回滚方案。
- 能从 RPO/RTO、权限和演练结果评审生产恢复能力。
