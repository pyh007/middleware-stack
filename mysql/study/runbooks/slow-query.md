# 慢 SQL 排查 Runbook

适用场景：接口延迟上升、数据库 CPU/IO 异常、慢日志增加、查询超时。目标是先控制影响、保存证据，再做可验证的最小修改。

## 1. 判断范围和紧急程度

- 记录开始时间、持续时间、受影响接口和租户、错误率与延迟分位数。
- 判断是单条 SQL、整体负载、锁等待、连接耗尽、磁盘还是复制问题。
- 如果正在造成严重影响，优先使用已经批准的限流、降级、熔断或终止异常查询流程。不要未经评估直接重启数据库。

## 2. 保存现场

本地实验可以使用：

```sql
SHOW FULL PROCESSLIST;

SELECT *
FROM performance_schema.events_statements_current
WHERE SQL_TEXT IS NOT NULL;

SELECT
    DIGEST_TEXT,
    COUNT_STAR,
    SUM_TIMER_WAIT,
    AVG_TIMER_WAIT,
    SUM_ROWS_EXAMINED,
    SUM_ROWS_SENT
FROM performance_schema.events_statements_summary_by_digest
ORDER BY SUM_TIMER_WAIT DESC
LIMIT 20;
```

生产采集时注意 SQL 参数、用户数据和日志的敏感性。至少保留：

- 完整 SQL 模板和代表性参数
- 调用频率、总耗时、平均及尾延迟
- 锁等待和错误信息
- 表结构、索引、表行数和统计信息
- `EXPLAIN` 计划及采集时间
- 当时 CPU、IO、连接、buffer pool 和复制状态

## 3. 分析执行路径

```sql
EXPLAIN FORMAT=TREE
SELECT ...;

-- 只有确认真实执行安全时才使用
EXPLAIN ANALYZE
SELECT ...;
```

依次检查：

1. 实际扫描行数是否远大于返回行数。
2. 估算与实际行数是否偏差很大。
3. 过滤条件是否可使用索引，是否存在函数或隐式转换。
4. 联合索引是否同时服务过滤、排序和分组。
5. 是否大量回表、额外排序或使用临时表。
6. 是否深分页、返回无用列或重复执行相同查询。
7. 时间是否主要消耗在锁、IO、网络或客户端，而不是执行计划。

## 4. 选择最小修改

候选方案包括改写谓词、减少返回列、游标分页、调整索引、分批处理、缓存或调整业务流程。每次尽量只改变一个核心变量，并写清为什么它能减少成本。

不要把 `FORCE INDEX`、扩大缓存、增加超时时间当作通用修复。它们可能掩盖统计信息、数据增长或查询设计问题。

## 5. 验证

- 结果行和业务语义与原 SQL 一致。
- 使用冷、热缓存和多组代表性参数测试。
- 比较实际行数、循环、延迟和资源，不只比较 `key` 名称。
- 压测写入开销、锁、磁盘、缓存和复制延迟。
- 检查数据增长后是否仍然稳定。

## 6. 上线和回滚

- 明确 DDL 算法、锁行为、临时空间、执行窗口和终止条件。
- 先灰度，观察吞吐、尾延迟、错误率和数据库资源。
- 保留原 SQL 或索引恢复步骤，确认回滚耗时。
- 上线后覆盖完整业务高峰，并把案例沉淀到 `mysql/study/cases/`。
