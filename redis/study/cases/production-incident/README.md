# Redis 生产事故综合案例：大促后的级联故障

## 场景

20:00 大促开始后，商品详情接口 P99 从 45ms 升至 4.2s，错误率 18%。Redis 主节点 CPU 92%，`used_memory` 接近 `maxmemory`，`evicted_keys` 快速增长，SLOWLOG 出现读取超大 Hash；应用连接数翻倍。20:02 一个热门商品缓存到期，数据库 QPS 从 8k 升至 46k。20:04 主节点重启，Sentinel 提升副本；应用有一部分仍写旧地址，库存缓存出现新旧值交替。值班同学提议立即 `FLUSHALL` 后全量预热。

## 已知约束

- 数据库安全水位为 15k QPS，超过后订单链路会受影响。
- 商品缓存可接受 60 秒陈旧，库存展示可接受 5 秒陈旧；扣库存以数据库为最终事实。
- Redis 为 1 主 2 从、3 Sentinel，AOF everysec；最近一次离线备份是 6 小时前。
- 变更窗口允许回滚最近发布，但不允许清空整个 Redis。

## 任务

1. 给出前五分钟止损顺序，并解释为什么不能执行 `FLUSHALL` 和全量并发预热。
2. 列出应用、Redis、Sentinel、数据库和主机侧要保存的证据，建立 20:00～20:04 时间线。
3. 区分热门缓存击穿、大 Key 阻塞、内存淘汰、重连风暴、旧主写入之间的因果关系。
4. 为商品详情和库存分别定义一致性边界、修复、灰度、验证和回滚。
5. 根据 AOF everysec 和异步复制说明可能的数据缺口，设计业务对账与恢复演练。
