# 综合案例参考处置

## 结论与前五分钟

先保护订单数据库和用户请求，再处理缓存正确性：限流/熔断低价值详情请求，启用 60 秒陈旧商品数据和请求合并，暂停非关键回填，限制 Redis 重试并确认客户端已跟随 Sentinel 新主。对热点商品定点续期或小批量预热，对超大 Hash 的危险命令做入口隔离。`FLUSHALL` 会把命中流量全部推向已超水位的数据库且删除无关数据，全量预热会形成第二次流量尖峰，因此都不可执行。

## 证据与因果

保存发布、流量、命中率、回源 QPS、连接池、超时和重试；保存 Redis `INFO` 各分区、SLOWLOG、LATENCY、热点/大 Key 样本、复制偏移；保存 Sentinel 事件和主地址变化；保存数据库 QPS、连接、慢查询；保存主机 CPU、网络和磁盘延迟。按统一时钟对齐后判断：热门键到期触发击穿，回源和回填加压；大 Key 命令阻塞事件循环；淘汰降低命中率形成正反馈；故障转移后的无限重试放大连接；旧拓扑缓存造成分叉写。

## 正确性与恢复

商品详情允许短时陈旧，可采用逻辑过期、互斥重建和 TTL 抖动。库存展示用版本号/更新时间拒绝旧回填，扣减仍以数据库事务为准，并用变更日志重新投递缓存。AOF everysec 和异步复制都不能承诺零丢失：从数据库订单、库存流水和事件日志确定权威区间，以故障时间和复制偏移圈定缺口，重放后抽样及总量对账。

## 验证与长期治理

灰度 1%→10%→50%→100%，每阶段观察错误率、P99、命中率、数据库 QPS、Redis CPU/内存、淘汰、连接和版本回退数。任一超过水位就回退策略并继续限流。长期拆分大 Key、建设热点检测、设置容量与连接预算、演练 Sentinel 客户端发现、明确 RPO/RTO，并把大促预热改成限速、可暂停、可回滚的任务。
