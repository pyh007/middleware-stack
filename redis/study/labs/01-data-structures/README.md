# 01 数据结构与业务建模

## 学习目标

- 区分对外数据类型与服务端内部编码，能解释编码转换的触发原因。
- 按访问、排序、去重、范围查询和消费语义选择类型，而不是照搬业务对象。
- 用 `TYPE`、`OBJECT ENCODING`、长度命令和 `MEMORY USAGE` 验证建模成本。

## 运行实验

先预测小/大 Hash、小/大 ZSet 分别采用什么编码，再运行：

```bash
make -C redis data-structures
```

## 核心机制

键空间字典把 Key 指向 Redis 对象，对象记录逻辑类型、编码和数据指针。String 可使用整数、短字符串或普通字符串表示；Hash、ZSet 等小集合倾向紧凑编码，超过配置阈值后转换为更适合查找或排序的结构。转换保持命令语义，却改变内存布局、遍历局部性和操作常数。因此“Hash 一定省内存”或“所有排行榜都用一个 ZSet”都缺少规模边界。

建模先写出读写路径：是否需要成员去重、按 score 范围取数、单字段更新、阻塞消费、修剪和过期。再限制单键元素数和响应大小，避免一个逻辑聚合变成大 Key。Cluster 中跨键原子操作还受槽位约束，相关对象是否使用 Hash Tag 需要和热点风险一起权衡。

## 观察证据与修改

实验输出 7 个对象的编码，断言 600 字段 Hash 与小 Hash、200 成员 ZSet 与小 ZSet 已发生编码转换；List 断言先进先出，并输出大 Hash 的实际内存。把 Hash 字段数从 600 改为 400，先根据 `hash-max-listpack-entries` 预测是否转换，再重跑并比较内存。

## 生产边界与完成标准

阈值调优会同时影响内存、CPU 和升级后的默认行为；大集合的删除、序列化、复制、AOF 重写和 Cluster 迁移都可能放大延迟。生产应对元素数、单键内存、最大响应和热度做分位数监控。
