# 01 参考答案

1. 结论：类型是 API 语义，编码是物理实现。Redis 根据 `hash-max-listpack-*` 等阈值从紧凑表示转换为哈希表；查找常数更稳定，但指针和桶增加内存。证据是实验中小/大 Hash 编码不同。生产不能把默认阈值当永久契约，应读取配置并用真实样本验证。
2. JSON String 一次读写简单、跨字段快照天然一致，但改单字段要整值序列化并传输；Hash 支持字段级原子命令和按需读取，但有字段开销且跨字段业务约束仍需事务/Lua。选择要同时验证对象大小、读写模式和版本演进。
3. ZSet 原生维护 score 顺序和范围查询；五千万成员会形成大 Key、单槽热点和昂贵迁移。可按业务窗口、地区或稳定哈希拆榜，另做全局 TopN 汇总，并限制返回范围。拆分会增加合并和一致性成本，需用压测验证。
4. List 能提供顺序弹出和阻塞等待，但缺少消费组、确认、重投和可观测位点。Stream 提供组和 Pending；需要跨分区持久消息、复杂重放时应选 Kafka 等系统。实验只证明顺序，不证明可靠队列能力。
5. 常见结果是整数 `int`、短字符串 `embstr`、长字符串 `raw`、小 Hash/ZSet 紧凑编码、大 Hash 为 `hashtable`、大 ZSet 为 `skiplist`。精确名称和阈值受版本、长度及配置影响，面试应说明边界。
6. 有效证据应包含阈值配置、前后元素数、编码和内存，而不只写“变快/变慢”。生产验证还要测 P99、复制流量、删除和重建时间。
