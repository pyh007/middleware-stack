# 02 过期淘汰与内存生命周期

## 学习目标

- 解释 `TTL=-1/-2`、惰性到期检查和后台主动采样的关系。
- 区分过期删除与 maxmemory 淘汰，能按数据价值选择策略。
- 把 Value、对象开销、碎片、复制/客户端缓冲纳入内存分析。

## 运行实验

```bash
make -C redis expiration-eviction
make -C redis eviction-pressure   # 显式压力实验：独立 8MiB 实例
```

第二个命令启动 `127.0.0.1:6389` 临时实例，写入约 10 MiB 数据触发 `allkeys-lru`，结束后自动删除；它不连接默认实例，也不属于 `all`。

## 核心机制

Redis 为带过期时间的键维护过期元数据。访问键时会检查并删除已到期对象，后台任务也会采样并在过期比例较高时继续扫描。因此过期时刻不是每个键精确同步释放的承诺。`TTL=-1` 表示键存在但没有过期，`-2` 表示键已不存在。

达到 `maxmemory` 后，Redis 根据 policy 选择候选键淘汰；候选可能只来自有 TTL 的键，也可能来自所有键。近似 LRU/LFU 是采样决策，不是全局精确排序。`used_memory` 还包括键对象、字典、过期表、复制积压、客户端缓冲等，进程 RSS 又会受到 allocator 碎片影响。

## 证据、修改与生产边界

安全实验断言 900ms 键到期、持久键和不存在键状态不同，并比较 64B 与 4KiB Value 的实际内存。压力实验断言 `evicted_keys` 增加且写入数大于存活数。把临时实例策略改为 `volatile-lru`，先预测无 TTL 键填满后的行为，再重跑。

生产切换策略前先盘点有 TTL/无 TTL 数据价值和源站承载力；大量同刻过期会带来删除 CPU 与回源峰值，应用应使用有界抖动并保护源站。监控 `used_memory/maxmemory`、RSS、碎片率、淘汰、过期、命中率和回源量。
