# 02 参考答案

1. `-1` 是键存在且永久，`-2` 是不存在。Redis 通过访问时检查与后台采样删除，不为每个键维护精确定时器；删除后 allocator 碎片也可能让 RSS 暂不归还。实验用 PTTL 和到期读证明逻辑消失，不能只看进程内存。
2. `volatile-lru` 只淘汰带 TTL 的键，可能在无 TTL 数据占满时无法释放；`allkeys-lfu` 可淘汰所有键，适合可重建缓存但可能误伤状态。选择前要给数据分级并压测回源，不能只依据命中率。
3. 原因可能是内存碎片、allocator 保留页或后台缓冲。联看 `used_memory_rss`、`mem_fragmentation_ratio`、数据集大小、AOF/复制/客户端缓冲和主机 RSS。直接重启虽可能降 RSS，却带来可用性与热缓存损失。
4. 同窗到期会增加主动删除开销，并把未命中并发推向数据库形成雪崩。可在业务最大有效期内增加固定种子的随机抖动、分批预热和源站限流；敏感会话不能随意延长，需按语义设上界。
5. 达上限后 LRU 近似淘汰，`evicted_keys` 增长，存活数小于写入数，`used_memory` 围绕上限波动。具体第几个键触发与对象开销、版本有关，实验断言行为而非固定数量。
6. `noeviction` 在无法分配时返回 OOM 类错误，应用应限流、降级或转权威存储，不能无限重试。生产变更 maxmemory/policy 前后都要验证命中率、错误率和源站水位，并准备恢复旧值。
