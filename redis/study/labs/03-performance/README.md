# 03 性能热点与可观测性

## 学习目标

- 解释单线程命令执行、网络往返、Pipeline 和队头阻塞之间的关系。
- 用原生信号区分热点、大 Key、慢命令、连接等待和持久化延迟。
- 建立“业务症状 → 分层证据 → 最小修复 → 验证回滚”的诊断顺序。

## 运行实验

```bash
make -C redis performance
```

先预测 600 次逐条 SET 与一个 Pipeline 的时间差，再说明为什么不能把一次本机结果当作生产 SLA。

## 核心机制

Redis 可以高效处理大量短命令，但每次客户端请求仍有网络、系统调用、解析和响应成本。Pipeline 把多条请求批量发送，主要减少往返；服务端仍按命令执行，并为批次缓存响应。批次过大可能增加客户端内存、队头阻塞和失败重试范围。

热 Key 是请求过度集中，会打满单核、网卡或分片；大 Key 是对象本身或单次返回过大，会放大序列化、阻塞、复制和迁移。诊断从业务错误率/P99/时间线开始，再看 `INFO clients/stats/memory/commandstats`、SLOWLOG、LATENCY、主机 CPU/网络/磁盘。不能用 `KEYS *` 或无界扫描制造二次事故。

## 观察证据与修改

实验校验两种写法都写入 600 个键，输出耗时对照；随后对一个键读 2000 次，创建 800 字段大 Hash，并输出 GET calls、`MEMORY USAGE`、blocked clients 和 slowlog 长度。把 Pipeline 批次拆为 10、50、200、600，预测吞吐和 P99 的变化，再在本机记录结果。

## 生产边界与完成标准

优化 Pipeline 同时要限制单批命令数/字节数和超时重试；拆热 Key 会增加聚合与一致性成本；拆大 Key 要处理双写、迁移和回滚。上线后以业务正确性、P99、吞吐、CPU、网络和内存共同验证。
