# 04 并发事务、Lua 与分布式锁

## 学习目标

- 复现读改写的丢失更新，解释命令原子不等于业务流程原子。
- 区分 MULTI/EXEC、WATCH 乐观重试与 Lua 原子脚本的保证和失败边界。
- 实现带 TTL、唯一令牌和原子比较删除的锁，并说明它仍非绝对互斥。

## 运行实验

```bash
make -C redis concurrency
```

## 核心机制

单个 Redis 命令原子执行，但 `GET` 后在客户端计算再 `SET` 是两个命令，中间可以被其他客户端插入。`MULTI/EXEC` 让排队命令连续执行，却不会自动根据旧值检查冲突，也不做数据库式回滚。`WATCH` 监控键版本，在 EXEC 前发生变化就让事务失败，客户端必须有界重试。

Lua 在一个 Redis 实例内原子执行，适合“检查并修改”，但脚本运行时阻塞其他命令，Cluster 多键要求同槽。锁用 `SET key token NX PX ttl` 获取，释放时 Lua 比较 token 后删除，防止超时后的旧持有者删掉新锁。进程暂停、续租失败、异步主从切换仍可能破坏互斥；不可重复外部副作用需要 fencing token 由下游拒绝旧持有者。

## 观察证据与修改

实验用 Barrier 让 8 个线程读到同一旧值，最终计数只为 1；WATCH 让 8×20 次更新得到 160，并输出冲突重试数；Lua 仅允许两次库存扣减；错误锁令牌不能删除锁。把线程数改为 16，先预测最终值、WATCH 重试和耗时，再重跑。

## 生产边界与完成标准

WATCH 高冲突时可能活锁，必须限次、退避并允许降级；Lua 要限制运行时间和 Key 数；锁 TTL 要覆盖操作并有超时观测，关键写还要幂等与 fencing。生产验证应注入超时、暂停、网络分区和故障转移。
