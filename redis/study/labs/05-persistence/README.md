# 05 持久化、重写与恢复窗口

## 学习目标

- 用 RPO/RTO 解释 RDB、AOF fsync 和混合持久化，而不是背诵“谁更安全”。
- 观察 BGSAVE、BGREWRITEAOF 的状态、fork/重写成本和失败信号。
- 通过 SIGKILL 与重启验证 AOF 数据卷恢复，同时保留明确的安全边界。

## 运行实验

```bash
make -C redis persistence
make -C redis crash-recovery   # 显式故障：默认 6379 短暂不可用
```

安全实验写入 100 个实验 Hash，执行后台快照与 AOF 重写后清理。故障实验写入一个 marker，等待 AOF 确认，对主容器发送 SIGKILL，保留命名卷、拉起服务、读取 marker 后删除；不会执行 volume delete，但会中断默认实例数秒，不属于 `all`。

## 核心机制

RDB 在某一时点生成快照，恢复读取快，但两次快照间的写可能丢失。AOF 记录写命令，`appendfsync everysec` 通常在吞吐与丢失窗口之间折中，但不能承诺零丢失。重写根据当前数据生成更紧凑 AOF，期间新写进入增量缓冲；重写成功不改变 fsync 本身的确认语义。

BGSAVE/重写通常需要 fork，写时复制会放大峰值内存，磁盘和页表规模会影响暂停。重启恢复时间还由文件大小、CPU、磁盘和校验修复决定。备份存在不代表可恢复，必须隔离加载并和业务权威源对账。

## 证据、修改与生产边界

实验断言 RDB/AOF 后台状态为 `ok`，输出快照耗时、AOF current/base size 和持久化配置；故障实验输出崩溃后恢复的精确键值。把写入数量改为 1000，预测 AOF 尺寸与重写耗时变化。生产需监控 fork 时间、COW 内存、RDB/AOF 状态、磁盘延迟与恢复演练的实际 RPO/RTO。
