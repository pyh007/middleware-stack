# 07 容量、安全、备份与生产运维

## 学习目标

- 用真实 `MEMORY USAGE` 样本估算数据集、余量、副本和后台任务峰值。
- 计算应用连接预算，理解连接池上限、maxclients 与重连风暴。
- 检查 ACL 身份并完成对象级 DUMP/RESTORE，建立恢复而非备份思维。

## 运行实验

```bash
make -C redis production-ops
```

## 核心机制

容量规划从真实类型、元素数、Key/Value 长度和 TTL 分布采样。序列化 Value 不包含 Redis 对象、字典、过期元数据、碎片、复制积压、客户端输出缓冲和 AOF/RDB 峰值。实验把 100 个 Hash 的平均内存外推到 100 万键，再加入 50% 余量和一个副本；这只是方法示例，生产要覆盖 P95/P99 大 Key 和增长周期。

连接预算是应用实例数 × 每实例池上限，加发布双倍实例、运维和复制连接，必须低于 maxclients。安全要组合网络隔离、TLS、ACL 最小权限、Secret 轮换和审计，本地默认用户与弱密码只用于学习。备份文件必须在隔离实例恢复、对账并测 RPO/RTO；DUMP/RESTORE 只证明对象级流程。

## 观察证据与修改

实验输出样本平均、容量外推、used_memory、连接数/maxclients、ACL 身份和池上限；随后 DUMP 一个 Hash、删除并 RESTORE，断言余额与版本恢复。把 payload 改为 1KiB，先预测容量估算比例；再把池上限乘以计划应用实例数，判断是否超过 70% 连接预算。

## 生产边界与完成标准

生产上线使用 [安全容量清单](../../runbooks/security-capacity.md)，恢复使用 [恢复 Runbook](../../runbooks/recovery.md)。任何 ACL/maxmemory/连接池变更都要灰度，监控正确性、P99、连接、内存、淘汰和复制，并保存旧配置回滚。

- [ ] 能写出带假设的容量和连接预算。
- [ ] 能解释对象恢复与全量恢复的差距。
- [ ] 能给出凭据轮换、最小权限、验证和回滚步骤。
