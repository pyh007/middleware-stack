# 08 缓存事故综合实战

## 学习目标

- 用可计数的后端调用区分穿透、击穿和雪崩，而不是混用术语。
- 验证空值缓存、互斥重建、TTL 抖动和版本化回填的效果与副作用。
- 把缓存治理连接到源站保护、业务正确性、监控、灰度和回滚。

## 运行实验

```bash
make -C redis incident-capstone
```

## 核心机制

穿透是查询不存在数据导致每次越过缓存；空值缓存或布隆过滤器可以挡住请求，但带来短时一致性窗口和维护成本。击穿是单个热点失效时并发回源；互斥重建、请求合并或逻辑过期能保护源站，但必须有超时、所有权和陈旧数据策略。雪崩是大量键同窗失效或节点故障导致回源尖峰；TTL 抖动只分散时间，不能替代源站容量、限流和降级。

数据库更新和缓存回填可能乱序，旧请求晚到会覆盖新值。用数据库版本/单调序列在 Lua 中比较，拒绝旧版本；扣库存等关键事实仍应由数据库事务、唯一约束和流水保障，缓存只做派生视图。

## 观察证据与修改

实验断言：不存在用户的后端访问从 8 降到 1；10 个并发热点回源从 10 降到 1；固定随机种子让 100 个 TTL 分散到至少 10 个桶、跨度至少 20 秒；version=1 被拒绝、version=3 被接受。把并发数改为 30，预测无锁/有锁回源数和延迟；把 TTL 抖动改为 0，观察桶数。

## 生产边界与完成标准

先保护数据库水位，再恢复缓存命中；禁止源站已过载时清库或全量并发预热。治理要监控按键并发、未命中原因、回源 QPS、重建等待、TTL 分布、版本倒退和业务错误。完成实验后闭卷处理 [综合生产事故](../../cases/production-incident/README.md)。
